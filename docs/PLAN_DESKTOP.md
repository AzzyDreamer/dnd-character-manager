# План: десктоп-сборка (Tauri), файловое хранилище и оффлайн-онлайн

> Статус: проектирование. Зафиксирован 2026-06-15 по итогам обсуждения релиза приложения
> как отдельного (десктопного) продукта.
> Этот документ **заменяет инфраструктурную часть** `PLAN_PARTY.md` (VPS + PocketBase + веб)
> на модель «десктоп без сервера». **Доменная модель партий** из `PLAN_PARTY.md`
> (роли ГМ/игрок, видимость `party/gm/hidden`, состав игрового лога, read-only чужие листы,
> отсутствие чата, ретенция) остаётся в силе и переиспользуется.

## Контекст

Приложение полностью клиентское: персонажи в `localStorage` одним ключом
(`src/utils/storage.ts`), доступ к хранилищу — только через хук `src/hooks/useCharacters.ts`.
На ветке `feat/character-store-seam` уже выделен async-интерфейс `CharacterStore`
(`list / upsert / remove / getActiveId / setActiveId`) и реализация `LocalCharacterStore`
поверх `localStorage` — это единственный шов для подмены хранилища. Сим в `main` ещё не влит.

Сборка — Vite, статика. Деплой на GitHub Pages уже настроен
(`base: '/dnd-character-manager/'` в `vite.config.ts`).

## Что меняется относительно PLAN_PARTY.md

| Аспект | PLAN_PARTY.md (отложено 2026-06-14) | Этот план |
|---|---|---|
| Форм-фактор | Веб + собственный VPS | Десктоп (Tauri) **и** веб (Pages) из одной кодовой базы |
| Хранилище | PocketBase / SQLite на сервере | Локальный файл (appData) на десктопе; `localStorage` в вебе |
| Онлайн | Обязательный сервер, доступен из браузера | Опционально, **только десктоп**, без своего сервера (LAN / VPN / P2P) |
| Расходы | VPS постоянно | $0 инфраструктуры |
| Доступ | Логин-гейт в онлайн-сборке | Без логина; онлайн — это «подключиться к хосту», а не «войти в сервис» |

**Переиспользуется без изменений** (доменные решения, не инфраструктура): роли ГМ/игрок;
видимость листа `party / gm / hidden`, живущая на членстве, а не на персонаже; состав
`game_events` (`roll / hp / rest / levelup / resource / system`); read-only открытие чужих
листов; отсутствие чата в v1; ретенция/экспорт лога. Меняется только **транспорт**
(PocketBase realtime → прямое соединение между десктопами) и **место хранения**.

## Зафиксированные решения (этот разговор)

| Вопрос | Решение |
|---|---|
| Десктоп-обёртка | **Tauri** (не Electron): системный WebView2 на Windows, бинарь ~единицы МБ, чистый фронтенд почти не трогает Rust |
| Веб-канал | GitHub Pages остаётся как есть, `localStorage`, без изменений |
| Шов хранилища | Существующий async `CharacterStore`; веб → `LocalCharacterStore`, десктоп → новый `FileCharacterStore` |
| Где хранит десктоп | Папка appData ОС (`appDataDir()` → `%APPDATA%\ru.azzydreamer.dndmanager\characters.json`; имя папки = identifier приложения); один файл, форма `CharacterStorage`, переиспользует `migrateCharacter` |
| Почему не «рядом с .exe» | Рассматривалось (портативно), но отклонено: appData переживает обновление/переустановку, всегда доступна на запись, не требует портативной сборки. «Спрятанность» лечится кнопкой «Открыть папку с данными» |
| Выбор реализации | Одна точка ветвления на старте: `const store = isTauri() ? fileStore : localStore` |
| `base` | Веб — `/dnd-character-manager/`, десктоп — `/` (Tauri отдаёт фронт с корня протокола); переключается режимом сборки `vite --mode desktop` |
| Упаковка | NSIS-установщик (per-user) как основной + опционально портативный zip; оба пишут в appData |
| Подпись кода | Отложена. Сейчас раздаём неподписанным (SmartScreen «Подробнее → Выполнить»). Позже — Azure Trusted Signing (~$10/мес) или SignPath (если OSS). Не блокер |
| Онлайн | Будущий этап, **только десктоп**, без своего сервера: хост слушает порт, гости подключаются по `IP:порт` (реальная LAN или виртуальная через Radmin/Tailscale/ZeroTier) |
| Веб и партии | Браузер не умеет хостить/принимать входящие → веб-пользователи в партии не входят. Осознанный размен ради $0 инфраструктуры |

## Архитектура сборки

Одна кодовая база, **три режима**, различаются флагом окружения и рантайм-проверкой:

```
            ┌─ веб (Pages)      vite (mode по умолч.)  base = /dnd-character-manager/,  store = Local
build ──────┼─ десктоп оффлайн  vite --mode desktop    base = /,                       store = File(appData)
            └─ десктоп онлайн   то же + ленивый импорт src/online/  (будущее)
```

- **Рантайм-различие** — `isTauri()` (проверка наличия Tauri API в окне). По нему выбирается
  стор и подключается онлайн-модуль.
- **Граница онлайна** — весь сетевой код лежит в `src/online/` и импортируется **только**
  динамически под `isTauri()`. Веб-бандл не должен содержать ни сетевого кода, ни Tauri-API.
  Это проверяется в CI (сборка обоих вариантов).
- `base` переключается режимом Vite (`vite --mode desktop`): десктопу нужен `/`, потому что
  Tauri отдаёт фронт с корня кастомного протокола, а не из подпапки. Tauri зовёт Vite сам через
  `beforeDevCommand`/`beforeBuildCommand` (`npm run dev|build -- --mode desktop`).

## Хранилище

Шов `CharacterStore` превращает «веб vs десктоп» в выбор реализации, а не в развилку по коду:

| Режим | Реализация | Где данные |
|---|---|---|
| Веб (Pages) | `LocalCharacterStore` (есть) | `localStorage`, ключ `dnd-characters` |
| Десктоп | `FileCharacterStore` (новый) | `<appData>/ru.azzydreamer.dndmanager/characters.json` |
| Десктоп онлайн (будущее) | `RemoteCharacterStore` / синк-обёртка над файлом | файл + сеть |

`FileCharacterStore`:
- читает/пишет один JSON той же формы `CharacterStorage` (`{ characters, activeCharacterId }`);
- «файла ещё нет» → пустой стор; переиспользует `migrateCharacter`;
- путь через Tauri (`appDataDir()` + `@tauri-apps/plugin-fs`), запись атомарная
  (во временный файл → rename), чтобы не побить данные при сбое на середине записи.

**Открытый вопрос (решить до D2):** один `characters.json` (зеркалит текущий стор, удобно
бэкапить) **или** файл на персонажа (удобно делиться/синхронизировать поштучно, ближе к
существующему per-character export). Дефолт — один файл.

## Упаковка и распространение

- **Tauri-таргеты (Windows):** `nsis` (per-user, в `%LOCALAPPDATA%`, без админа) — основной;
  портативный zip с .exe — опционально. Данные в appData → обновление установщика их не трогает.
- **WebView2:** на Win11 предустановлен; на Win10 — почти везде (через Edge). Tauri может
  приложить bootstrapper в установщик на случай отсутствия.
- **Подпись:** сейчас нет. В доку релиза — заметка «как запустить неподписанное» для пользователя.
- **macOS/Linux:** вне области v1 (Windows-first). Архитектура кроссплатформенна, но подпись
  и нотаризация под другие ОС — отдельный разговор.

## Онлайн (будущее, только десктоп) — без своего сервера

Заменяет серверную модель `PLAN_PARTY.md`. Доменная часть (партии, видимость, лог, read-only)
переносится почти дословно; меняется транспорт.

- **Топология:** один игрок хостит (десктоп биндит TCP/WebSocket-порт через Rust-сторону,
  где нет CORS и есть входящие соединения), остальные подключаются по `IP:порт`.
- **Через интернет без проброса портов:** виртуальная локалка — Radmin VPN / Hamachi /
  Tailscale / ZeroTier. Приложению интеграция не нужна — нужно лишь **уметь коннектиться
  к произвольному `IP:порт`** и **не зависеть только от авто-обнаружения** (broadcast/mDNS
  часто не ходит через VPN-локалки; ручной ввод адреса работает всегда).
- **Синхронизация:** LWW из `PLAN_PARTY.md` рассчитан на авторитетный сервер; для P2P модель
  синка проектируется отдельно на этом этапе (вероятно «хост — источник истины для сессии»).
- **Что НЕ нужно вообще:** VPS, PocketBase, логин-гейт, публичная регистрация — всё это
  отменяется вместе с серверной моделью.

## Дорожная карта

Ближнее (D0–D4) конкретно; онлайн (D5+) — большой отдельный этап на потом.

### D0 — Слить async-сим в `main` *(предусловие, делает владелец)*
`feat/character-store-seam` → `main`. Без него `FileCharacterStore` не на что вешать.
Каркас Tauri (D1) от сима **не зависит** и может идти параллельно.

### D1 — Каркас Tauri
1. `src-tauri/` (конфиг, иконки, метаданные), `@tauri-apps/cli`, `@tauri-apps/plugin-fs`.
2. Скрипты `tauri dev` / `tauri build`; условный `base` по `VITE_TARGET`.
3. Критерий: `npm run dev` (веб) и `tauri dev` (десктоп) оба стартуют; `npm run build`
   веб-режима не изменился; десктоп показывает текущее приложение в окне.

### D2 — FileCharacterStore *(после D0)*
1. `FileCharacterStore implements CharacterStore` поверх appData-файла, атомарная запись.
2. Точка выбора стора на старте по `isTauri()`.
3. Критерий: десктоп сохраняет персонажей в `<appData>/.../characters.json` и переживает
   перезапуск; веб по-прежнему на `localStorage`; импорт/экспорт JSON работают в обоих.

### D3 — UX и упаковка десктопа
1. Кнопка «Открыть папку с данными» (Tauri opener) в настройках.
2. Иконка, имя окна, метаданные сборки.
3. Сборка `nsis` (per-user) + портативный zip; заметка про неподписанный запуск.
4. Критерий: установщик ставит приложение без админа, данные ложатся в appData,
   деинсталляция не трогает данные (или предлагает удалить — решить).

### D4 — Граница онлайн-модуля (заглушка)
1. Пустой `src/online/` за ленивым импортом под `isTauri()`; точка инъекции стора готова
   принять `RemoteCharacterStore`.
2. Критерий: веб-бандл не содержит `src/online/` и Tauri-API (проверка размера/импортов в CI).

### D5+ — Онлайн-партии *(большой этап, потом)*
Транспорт (хост слушает порт, коннект по `IP:порт`, ручной ввод адреса), сессионная модель,
перенос доменной части `PLAN_PARTY.md` (членство и видимость, read-only листы, игровой лог,
ретенция/экспорт) поверх P2P вместо PocketBase. Проектируется отдельным документом, когда дойдём.

## Риски и тонкие места

- **`base` обязан быть условным**: ошибёшься — на одном из таргетов ассеты 404 (Pages-подпапка
  vs кастомный протокол Tauri).
- **Утечка онлайн/Tauri-кода в веб-бандл**: только ленивые импорты под `isTauri()`, иначе
  браузерная сборка либо распухнет, либо упадёт на отсутствующем Tauri API. Страховка — CI на оба варианта.
- **Атомарность записи файла**: писать во временный и переименовывать, иначе сбой на середине
  бьёт `characters.json`. Желателен `.bak` на одну версию назад.
- **appData переживает удаление приложения**: решить, чистим ли данные при деинсталляции
  (по умолчанию — оставлять, предлагать удалить).
- **P2P-синк сложнее серверного**: LWW из PLAN_PARTY не переносится «как есть» на десктоп-хост —
  модель проектируется в D5.
- **Тулчейн Tauri в CI**: для сборок нужен Rust + системные build-зависимости на раннере.
- **Лицензия на контент**: данные D&D выходят за SRD (Grim Hollow, сеттинги, полные тексты).
  Для личной/узкой раздачи — ок; **публичный** релиз/OSS/монетизация — отдельный юридический
  трек, не решается кодом. Это блокер именно публичной раздачи, не самой сборки.

## Не делаем сейчас

- Реальный сетевой код (только граница-заглушка в D4).
- macOS/Linux сборки, подпись кода, авто-обновление.
- Публичный релиз/OSS до решения вопроса лицензии на контент.
