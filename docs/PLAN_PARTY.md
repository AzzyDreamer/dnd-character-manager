# План: party-функционал (онлайн-партии, листы сопартийцев, игровой лог)

> Статус: не начат. Зафиксирован 2026-06-13 по итогам обсуждения деплоя и онлайн-направления.
> Бэкенд: PocketBase на собственном VPS. Чата НЕ будет — только автособираемый игровой лог
> (броски + события листа). Регистрация закрытая (аккаунты создаёт админ вручную),
> инвайты от игроков — отдельный будущий этап.

## Контекст

Сейчас приложение полностью клиентское: персонажи лежат в `localStorage` одним ключом
(`src/utils/storage.ts`), компоненты ходят к хранилищу только через хук
`src/hooks/useCharacters.ts` — это единственный шов для подмены на сетевое хранилище.
Бросалка кубов централизована в `src/components/DiceRollProvider.tsx` (`roll()` /
`openConfig()`), консольные броски — в `DiceTab.tsx`. Навигация — hash-роутинг в
`App.tsx` (`#/characters`, `#/sheet/<id>`, ...), туда же встанут экраны партий.

Деплой: статика собирается Vite и заливается на VPS (nginx); GitHub Pages остаётся
параллельным «локальным» развёртыванием без бэкенда.

## Зафиксированные решения

| Вопрос | Решение |
|---|---|
| Бэкенд | PocketBase (один бинарь: auth + SQLite + realtime SSE + файлы), рядом со статикой на том же VPS |
| Чат | Нет. Свободного пользовательского текста в v1 нет вообще — только структурированные события |
| Шифрование | TLS в транспорте; E2E не делаем (нужна читаемость логов для модерации). At-rest — опционально, не в v1 |
| Регистрация | Закрытая: `users.createRule = null`, аккаунты создаёт админ в админке PocketBase |
| Доступ к приложению | В онлайн-режиме весь функционал за логином. Сборка без `VITE_BACKEND_URL` = текущий локальный режим (для GH Pages) |
| Кто создаёт партии | Любой авторизованный; создатель автоматически мастер (ГМ) |
| Видимость листа | Выбирается игроком при привязке персонажа к партии («спрашивать при вступлении»): вся партия / только ГМ / никто |
| Состав игрового лога | Броски кубов + события листа (HP, отдыхи, левел-ап, ресурсы). Заметок мастера в v1 нет |
| Realtime | Всё живое: лента событий и открытые чужие листы обновляются по подписке |
| Ретенция лога | Автоудаление событий старше N дней (конфиг, по умолчанию 90) + ручной экспорт ленты в файл |
| Юр. гигиена | Экран правил при первом входе (фиксация `tosAcceptedAt`), поле `suspended`, поле `invitedBy` про запас |

## Цели

1. Игрок привязывает персонажа к партии и явно выбирает, кто видит его лист.
2. Сопартийцы и ГМ открывают чужие листы строго read-only; открытый чужой лист
   обновляется живьём.
3. У ГМ есть сводный вид партии: карточки персонажей с ключевыми статами (HP/КД/состояния),
   обновляющиеся живьём.
4. Лента партии автоматически собирает броски и события листов всех участников,
   обновляется мгновенно, листается в прошлое, экспортируется в файл.
5. Всё это не ломает локальный режим: сборка без бэкенда работает как сейчас.

## Не-цели (v1)

- Чат и любой свободный текст между пользователями (включая заметки мастера) — позже,
  вместе с механикой жалоб/модерации.
- Инвайты от игроков, лимиты инвайтов, цепочка ответственности — отдельный этап после v1.
- Совместный боевой трекер/инициатива, шёпоты, тайные броски «только ГМ видит результат».
- E2E-шифрование.
- Публичная регистрация.

## Архитектура

```
VPS
├── nginx (TLS)
│   ├── /            → статика dist/ (фронт)
│   ├── /api/  /_/   → 127.0.0.1:8090 (PocketBase: API + админка)
├── pocketbase (systemd, версия бинаря зафиксирована)
│   └── pb_data/     → SQLite + загруженные файлы (портреты); бэкап tar по крону
```

Фронт определяет режим на этапе сборки: `import.meta.env.VITE_BACKEND_URL` задан →
онлайн-режим (логин-гейт, сетевое хранилище, партии); не задан → локальный режим,
код партий не активируется. CI собирает и проверяет оба варианта.

## Схема данных (коллекции PocketBase)

Правила записываются декларативно в PocketBase; ниже — смысл, не дословный синтаксис.

### users (auth-коллекция)
- `displayName` text, `suspended` bool, `tosAcceptedAt` date, `invitedBy` relation→users (пока пустое).
- create: только админ. view/list: любой авторизованный (сервис закрытый и маленький;
  нужно для выбора участников ГМом). update: сам пользователь (только displayName/tosAcceptedAt).

### characters
- `owner` relation→users, `data` json (объект `Character` целиком, как в localStorage),
  `portrait` file (заменяет base64 `portraitDataUrl`), `schemaVersion` number.
- CRUD: владелец. view дополнительно: участник/ГМ партии, где есть `party_members`
  с этим персонажем и подходящей `visibility` (см. ниже).

### parties
- `name` text, `gm` relation→users, `description` text (опц.).
- create: любой авторизованный, при этом `gm = @request.auth.id` (правило проверяет).
- update/delete: ГМ. view/list: ГМ + участники.

### party_members
- `party` relation→parties, `user` relation→users,
  `character` relation→characters (nullable — участник добавлен, но персонажа ещё не привязал),
  `visibility` select: `party` | `gm` | `hidden`.
- create: ГМ партии (добавляет пользователя, `character = null`).
- update: сам участник (привязка персонажа + visibility — диалог «вступления»), ГМ — только delete.
- view/list: участники партии и ГМ.
- Видимость живёт здесь, а не на персонаже: один персонаж в разных партиях может
  показываться по-разному.

### game_events
- `party` relation, `actor` relation→users, `character` relation (nullable),
  `type` select: `roll` | `hp` | `rest` | `levelup` | `resource` | `system`, `payload` json.
- create: участник партии, `actor = @request.auth.id`. view/list: участники + ГМ.
  update: никто. delete: ГМ (модерация).
- `payload` по типам:
  - `roll`: `{ expression, total, breakdown, label? }` (из `DiceRollResult`);
  - `hp`: `{ delta, from, to, temp? }`;
  - `rest`: `{ kind: 'short' | 'long' }`;
  - `levelup`: `{ class, level }`;
  - `resource`: `{ key, delta }`.
- Ретенция: cron-hook в `pb_hooks` удаляет события старше N дней (конфиг, дефолт 90).

## Этапы

### Этап 0 — async-шов хранилища (без сервера, поведение не меняется)

1. `src/utils/storage.ts`: выделить интерфейс `CharacterStore` с **async** методами
   `list / upsert / remove / getActiveId / setActiveId`; текущая логика становится
   `LocalCharacterStore`. Существующие export/import-функции не трогаем.
2. `useCharacters`: перевести на async (флаг `loading` уже есть), обновить вызовы
   в компонентах (`CharacterList`, `CharacterCreator`, `CharacterSheet`, импорт в `App.tsx`).
3. Критерий: приложение работает как раньше, `npm run build` обоих режимов зелёный.

### Этап 1 — сервер

1. PocketBase на VPS: systemd-юнит, nginx-проксирование `/api/` и `/_/`
   (админку дополнительно ограничить — хотя бы нестандартным путём/allowlist IP).
2. Коллекции и правила по схеме выше; `users.createRule = null`.
3. Бэкап `pb_data` по крону (tar + ротация). Версию бинаря записать в доку деплоя.
4. Критерий: создание пользователя в админке → логин через API → CRUD своего
   персонажа работает, чужого — запрещён (проверить руками через консоль).

### Этап 2 — auth на фронте

1. SDK `pocketbase`, модуль `src/online/client.ts` (singleton, инициализация по
   `VITE_BACKEND_URL`; в локальном режиме не импортируется динамически).
2. Логин-гейт в `App.tsx` (онлайн-режим): экран входа до любого контента;
   `suspended` → принудительный выход.
3. Экран правил при первом входе: текст правил (логи хранятся и читаемы администратором,
   срок хранения, запрещённое), кнопка «принимаю» → `tosAcceptedAt`.
4. Все новые строки — через i18next (en + ru), новые namespace-ключи.
5. Критерий: онлайн-сборка не показывает ничего до логина; локальная сборка
   не содержит логина вовсе.

### Этап 3 — облачные персонажи

1. `RemoteCharacterStore` поверх коллекции `characters`: list по owner,
   upsert с LWW-разрешением конфликтов по `updatedAt` (перед записью сравнить;
   проигравшая версия логируется в консоль), remove.
2. Миграция при первом входе: если в localStorage есть персонажи — предложить
   залить в облако (повторно использовать `migrateCharacter`); localStorage после
   успешной заливки очищается с подтверждением.
3. Портреты: при сохранении `portraitDataUrl` конвертируется в Blob → file-поле
   `portrait`; в `data` остаётся только ссылка/маркер. Рендер поддерживает оба
   варианта (старые data-URL продолжают работать до первого пересохранения).
4. Добавить `schemaVersion` в `Character` (текущая = 1) и проставлять в миграции.
5. Критерий: один аккаунт на двух устройствах видит одних и тех же персонажей;
   конфликт двух правок решается в пользу свежей без порчи данных.

### Этап 4 — партии: членство и экраны

1. Роуты `#/parties` (список + создание) и `#/party/<id>` (экран партии) в hash-роутер
   `App.tsx`; пункт в `TopNavBar` (онлайн-режим).
2. Создание партии: имя → создатель становится ГМ.
3. ГМ добавляет участников (выбор из списка пользователей по displayName/email).
4. Диалог вступления у участника: выбор персонажа из своих + выбор `visibility`
   (вся партия / только ГМ / никто) — это и есть настройка приватности.
5. Экран партии: список участников с карточками персонажей (имя/класс/уровень/портрет),
   плейсхолдер ленты событий.
6. Критерий: полный цикл «создал партию → добавил игрока → игрок привязал персонажа
   с выбранной видимостью» двумя аккаунтами.

### Этап 5 — чужие листы read-only + realtime листов

1. `CharacterSheet` получает режим `readOnly`: блокируются все мутации. Основной
   механизм — заглушка вместо `updateCharacter` + скрытие явных контролов
   редактирования. Инвентаризация точек мутации — первая задача этапа (лист большой:
   HP-контролы, инвентарь, слоты заклинаний, ресурсы, отдыхи, трансформации, заметки).
2. Открытие чужого листа с карточки участника — по правилам видимости (сервер всё
   равно не отдаст запрещённое; UI просто не показывает кнопку).
3. Realtime: подписка на запись `characters/<id>` открытого чужого листа
   (`subscribe` → обновление стейта); отписка при закрытии.
4. Сводный вид ГМ: сетка карточек «HP / КД / состояния / активные эффекты» по всем
   видимым персонажам партии, подписка на коллекцию с фильтром по партии.
5. Критерий: правка листа игроком видна ГМ без перезагрузки; `visibility=hidden`
   не показывается даже ГМ (только факт участия); ни одна мутация из чужого листа
   не уходит на сервер.

### Этап 6 — игровой лог

1. Модуль `src/online/gameLog.ts`: `logEvent(type, payload)` — очередь с отправкой
   в `game_events`; без сети/в локальном режиме — no-op (события не копятся бесконечно).
2. Понятие «активная партия» персонажа: если персонаж привязан к партии, события его
   листа идут в её лог. Тумблер «приватный бросок» в бросалке (не логировать конкретный бросок).
3. Точки эмита:
   - броски: `DiceRollProvider.roll()` / результат `openConfig`-меню / консоль `DiceTab`;
   - события листа: обработчики урона/лечения, короткого/длинного отдыха, левел-апа,
     трат ресурсов в `CharacterSheet` (инвентаризация хендлеров — задача этапа;
     эмитим из явных обработчиков, НЕ диффом состояния — дифф хрупок).
4. Лента на экране партии: realtime-подписка + пагинация назад, фильтры по
   типу/персонажу, локализованный рендер каждого типа события.
5. Ретенция: cron-hook в `pb_hooks` (JS) — удаление старше N дней; N в конфиге хука.
6. Экспорт ленты в `.txt`/`.json` кнопкой на экране партии (для «логов в файловой
   системе» у мастера; серверные данные и так в `pb_data`).
7. Критерий: бросок на листе игрока появляется в ленте у ГМ мгновенно; приватный
   бросок не появляется; экспорт скачивает читаемый файл.

### Этап 7 — после v1 (зафиксировано, не делаем сейчас)

- Инвайты: `invitedBy`-цепочка, лимит инвайтов, регистрация по коду (hook).
- Свободный текст (заметки мастера) + кнопка «пожаловаться» + коллекция `reports` —
  только вместе.
- At-rest шифрование payload'ов, если появится чувствительный текст.
- Шёпоты/тайные броски, боевой трекер.

## Риски и тонкие места

- **Этап 0 шире, чем кажется**: вызовы `addCharacter`/`updateCharacter` сейчас
  синхронные с try/catch; проверить каждое место на гонки (двойное сохранение,
  сохранение при размонтировании).
- **Read-only лист — самый трудозатратный UI-кусок** (этап 5.1): мутации размазаны по
  большому листу. Заглушка на уровне `updateCharacter` страхует от пропущенных контролов.
- **LWW достаточен, но груб**: одновременная правка с двух устройств теряет одну из
  правок целиком. Для одного игрока это редкий случай; осознанно принимаем в v1.
- **GH Pages не должен сломаться**: онлайн-код подключать через ленивые импорты по
  флагу окружения; в CI собирать оба варианта.
- **i18n**: каждый новый экран — en + ru сразу, иначе разъедется (см. I18N_CONTEXT.md).
- **PocketBase 0.x**: фиксировать версию бинаря, обновлять только по changelog.
- **Квота localStorage и портреты** уходят как проблема после этапа 3 (портреты — файлы).

## Порядок и независимость

Этапы 0→1→2→3 строго последовательны (каждый полезен сам по себе: 0 — рефакторинг,
1–2 — закрытый вход, 3 — синк между устройствами). Этапы 4→5→6 последовательны между
собой. Деплой-workflow (rsync по SSH из GitHub Actions) независим и может быть сделан
в любой момент.
