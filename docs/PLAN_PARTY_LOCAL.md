# План: локальные party (LAN/VPN, P2P, без сервера)

> Статус: проектирование. Зафиксирован 2026-06-16.
> Это конкретизация этапа **D5+** из `docs/PLAN_DESKTOP.md`: онлайн-партии **только на десктопе, без своего сервера** —
> один игрок хостит, остальные подключаются по `IP:порт` (реальная LAN или виртуальная через Radmin/Tailscale/ZeroTier).
> **Доменная модель** (роли, видимость, игровой лог, read-only листы, отсутствие чата) переиспользуется из
> `docs/PLAN_PARTY.md`; **транспорт** меняется: PocketBase-realtime → прямое соединение между десктопами.

## Контекст и что переиспользуется

- Персонажи лежат локально (`FileCharacterStore`, appData) — party это **не меняет**. Партия — отдельный
  live-слой поверх: игрок **транслирует снимок** своего листа другим, а канонический персонаж остаётся у него локально.
  `RemoteCharacterStore` (облачный стор из старого плана) здесь НЕ нужен.
- Из `PLAN_PARTY.md` берём без изменений: роли **ГМ/игрок**; видимость листа **`party / gm / hidden`** (выбор при
  вступлении); состав игрового лога (`roll / hp / rest / levelup / resource / system`); **read-only** открытие чужих
  листов; **нет чата** в v1; ретенция/экспорт лога.
- Точки эмита событий уже централизованы: броски — `DiceRollProvider` / `DiceTab`; события листа (HP, отдыхи,
  левел-ап, ресурсы) — обработчики в `CharacterSheet`.

## Ключевые технические решения

| Решение | Обоснование |
|---|---|
| **Вся сеть — на Rust** (tokio), фронт общается через команды/события Tauri | Вебвью в secure-context (`https://tauri.localhost`) → `ws://` из JS блокируется как mixed-content. Rust этим правилам не подчиняется. |
| **Транспорт — WebSocket** поверх LAN/VPN | Кадрирование + простые JSON-сообщения; работает по виртуальной локалке без signaling. WebRTC (для интернета без VPN) — post-v1. |
| **Топология «звезда»: хост = хаб** | Хост держит состояние партии и **ретранслирует** снимки/события всем. Проще полного mesh, достаточно для домашней группы. |
| **Хост авторитетен для сессии** | Источник истины: состав, лента, актуальные снимки. Игрок владеет своим персонажем (пушит апдейты), хост хранит/раздаёт. |
| **Подключение — ручной ввод `IP:порт`** + **код партии** | Не полагаемся на broadcast (через VPN часто не ходит). Код партии — деттерент против случайных подключений на общей VPN (как UA-гейт: не настоящая авторизация, а отсечь чужих). |

## Архитектура

```
ХОСТ (десктоп ГМа)                          КЛИЕНТ (десктоп игрока)
┌───────────────────────────┐               ┌───────────────────────────┐
│ webview (React)           │               │ webview (React)           │
│   src/online/party.ts     │               │   src/online/party.ts     │
│      ↕ invoke / events     │               │      ↕ invoke / events     │
│ Rust: party server (tokio)│◀── WS:port ───│ Rust: party client (tokio)│
│   bind 0.0.0.0:<port>     │   (LAN/VPN)    │   connect ws://host:port  │
│   состояние партии + лог   │               │                           │
└───────────────────────────┘               └───────────────────────────┘
```

- **Rust-модуль** (`src-tauri/src/party/`): хост поднимает WS-сервер (`tokio` + `tokio-tungstenite`), клиент — WS-клиент.
  Команды Tauri: `party_host_start(port, code) / party_join(addr, code, name) / party_send(msg) / party_leave()`.
  События в фронт: `party://message`, `party://presence`, `party://state`, `party://error`.
- **Фронт** (`src/online/`, ленивый импорт под `isTauri()` — это и есть граница из D4, которую делаем здесь):
  тонкий клиент над командами/событиями + party-UI + интеграция (снимки листа, эмит событий, read-only рендер).
- **Веб-бандл не трогаем**: весь party-код за `isTauri()` + динамический импорт; в CI проверяем, что `@tauri-apps`/online
  не попали в веб-сборку (как уже сделано для `FileCharacterStore`).

## Протокол (JSON-сообщения поверх WS)

Версионируется (`v` в каждом сообщении) — разные версии приложения должны явно отбиваться при рукопожатии.
- `hello` (клиент→хост): `{ code, displayName, appVersion }` → хост проверяет код/версию, отвечает `welcome` или `reject`.
- `member-snapshot` (игрок→хост→все по видимости): read-only снимок листа (`{ characterId, visibility, data }`).
- `event` (любой→хост→все): `{ type: roll|hp|rest|levelup|resource|system, payload }` — для ленты.
- `presence` (хост→все): список участников (online/offline, привязанный персонаж, видимость).
- `party-state` (хост→все): имя партии, ГМ, состав.
- `bye` / разрыв: хост обновляет presence; у игрока — пометка «оффлайн».

## Дорожная карта

### LP0 — транспорт-фундамент ✅ реализовано + проверено (рукопожатие зелёное на реальных сокетах)
1. `src-tauri/src/party/` — WS-сервер (хост) и WS-клиент на tokio; команды/события Tauri; рукопожатие с кодом партии и
   проверкой версии протокола. Капабилити под новые команды.
2. `src/online/` (ленивый, под `isTauri()`) — обёртка над командами/событиями.
3. UI «создать партию» (порт + код, показ своего `IP:порт`) и «подключиться» (ручной ввод `addr` + код + имя).
4. Критерий: два экземпляра приложения соединяются по `IP:порт`, проходят рукопожатие, обмениваются `hello/welcome`;
   неверный код/версия → отказ; веб-сборка чистая (нет online/@tauri-apps).

**Как сделано (LP0):**
- Rust: `src-tauri/src/party/{mod,protocol,host,client}.rs`. Команды `party_host_start(port, code) → {port, ips}`,
  `party_join(host, port, code, displayName)`, `party_send(msg)`, `party_leave()`. События `party://{status,peer,message,error}`.
  Зависимости: `tokio`, `tokio-tungstenite`, `futures-util`, `if-addrs`. Сеть крутится на `tauri::async_runtime` (tokio).
- App-команды Tauri v2 **не требуют записей в capabilities** (ACL гейтит только плагины/`core:`); `listen` покрыт
  `core:event:default` из `core:default`. Имена событий с `://` проходят валидатор Tauri (разрешены `:` и `/`).
- Код партии = `base64url("ip:port|secret")`, кодируется/декодируется в `src/online/protocol.ts` (Rust видит только секрет).
  `PROTOCOL_VERSION = 1`, `DEFAULT_PORT = 47331` — синхронны между Rust и TS.
- Фронт: `src/online/{protocol,party}.ts` + `PartyPanel.tsx` (ленивый импорт из `App.tsx` под `isTauri()`); вкладка
  «Партия» в навбаре и маршрут `#/party` только под Tauri (в вебе схлопывается в home).
- Чистота веб-бандла: `scripts/check-web-imports.mjs` запрещает статический импорт `@tauri-apps`/`src/online`/desktop-онли
  из обычных файлов; шаг в `deploy-pages.yml`. Проверено: `party_*` лежат в отдельном `PartyPanel-*.js`, в `index-*` их нет.
- Сетевой слой развязан от Tauri через трейт `PartyEvents` (host/client — generic; в проде сток = `AppHandle`, в тестах —
  записывающий мок). Это дало интеграционные тесты транспорта на **реальных loopback-сокетах** (`cargo test party`,
  `--test-threads=1`, всё на `tauri::async_runtime::block_on` ради единого реактора): welcome+peer-join, reject `bad-code`,
  reject `version-mismatch` — все зелёные.
- Проверки пройдены: `cargo check`, `cargo test` (3/3), `tsc --noEmit`, `npm run build`, lint новых файлов.
  **GUI двумя инстансами** не гонялся агентом: в песочнице агента межпроцессный loopback недостоверен (см. заметку про
  редирект AppData/MSIX), поэтому два окна надёжнее проверять в реальной среде пользователя. Код рукопожатия доказан тестами.

### LP1 — членство и presence ✅ реализовано (роли + ростер; тесты на сокетах зелёные)
1. Создание партии (хост = ГМ), вступление (имя + код), список участников, presence (online/offline), выход/разрыв.
2. Экраны `#/party` (создать/подключиться) и `#/party/active` в hash-роутер; пункт в `TopNavBar` под `isTauri()`.
3. Критерий: полный цикл «хост создал → игрок подключился → виден в составе → отключился» двумя экземплярами.

**Как сделано (LP1):**
- Хост держит состав партии (`Room` в `host.rs`): ГМ + игроки по id подключения; источник истины о presence. Любая правка
  (вход/выход) → `broadcast_state`: рассылка снимка `party-state` всем клиентам + локальный `party://state` для UI хоста.
- Сообщение `party-state` (хост→все): `{ partyName, gmId:"gm", members:[{id,displayName,role:gm|player,online}] }`.
  Клиент в `client.rs` отделяет его от прочих кадров и эмитит `party://state` (иначе — `party://message`).
- `party_host_start` теперь принимает `displayName` (имя ГМ) + `partyName` (пустое → дефолт по имени ГМ).
- Отключение игрока → `online:false` (не исчезает из состава) + ребродкаст. Стабильная идентичность при реконнекте и
  чистка stale-offline — сознательно отложены в LP4.
- Фронт: `PartyPanel` показывает **ростер для обоих ролей** (корона=ГМ, online-точка, «(вы)»), имя партии в заголовке;
  форма хоста спрашивает имя ГМ + имя партии. Типы/обёртка — `party.ts` (`PartyMember`, `PartyStateSnapshot`, `onState`).
- Тесты: добавлен `roster_broadcast_reaches_client_and_host` (клиент и хост видят ГМ+игрока на реальных сокетах).
  `cargo test party` — 4/4 зелёные; `tsc`, `npm run build`, purity, lint — зелёные.
- **Отложено в LP1:** отдельный sub-маршрут `#/party/active` (активная сессия живёт внутри `PartyPanel`, без своего URL —
  браузерный Back из активной партии пока не возвращает на меню). Низкий приоритет, не влияет на критерий.

### LP2 — снимки листов + read-only ✅ реализовано (транспорт+видимость+сводка+полный read-only лист)
1. Игрок привязывает персонажа к партии и выбирает видимость (`party/gm/hidden`); при изменении листа шлётся снимок.
2. **Read-only режим `CharacterSheet`** — самый трудозатратный кусок (мутации размазаны по листу). Основной механизм —
   заглушка на `updateCharacter` + скрытие контролов; первая задача этапа — инвентаризация точек мутации.
3. Открытие чужого листа с карточки участника по правилам видимости; живое обновление по приходящим снимкам.
4. Сводка ГМа: сетка карточек (HP/КД/состояния/активные эффекты) по видимым персонажам.
5. Критерий: правка листа игроком видна ГМу без перезагрузки; `hidden` не показывается даже ГМу; из чужого листа
   ни одна мутация не уходит.

**Как сделано (LP2, часть 1 — транспорт + видимость + сводка):**
- **Инвентаризация мутаций листа сделана:** в `CharacterSheet` все мутации идут через ЕДИНСТВЕННЫЙ проп `onUpdate`
  (протянут в ~13 секций: Inventory/Spells/Roleplay/DeathSaves/Conditions/Concentration/Senses/Movement/Resistances/
  Transformation/ActiveForms/Kindred/WildShape). ⇒ заглушка `onUpdate=no-op` — гарантированный предохранитель read-only.
  Плюс безопасность архитектурная: снимки **однонаправленные** (владелец→зрители), upstream-канала правки чужого листа нет.
- **Снимки листов:** игрок шлёт `member-snapshot{characterId,characterName,visibility,data}` → хост ретранслирует
  `party-snapshot` по видимости: `party` — всем кроме автора + ГМ; `gm` — только ГМ (локальный эмит); `hidden` — снятие
  (`data:null`) у всех. Хост хранит последние снимки и **догоняет** поздно подключившихся (party-видимые). Команда
  `party_share`; событие `party://snapshot`. ГМ тоже может делиться (from=`"gm"`).
- **Привязка на уровне App** (`partyBinding` + эффект): снимок уходит при правке листа на ЛЮБОМ экране (PartyPanel может
  быть размонтирован) — это и закрывает «правка видна ГМу без перезагрузки». Входящие снимки (`partySnapshots`) тоже копятся
  в App, переживая уход с экрана партии. Обёртка — `party.ts` (`shareCharacter`/`unshareCharacter`/`onSnapshot`).
- **UI:** в `PartyPanel` — выбор «каким персонажем делюсь» + видимость, и **сводка ГМа** (карточки HP/КД/состояния по
  видимым листам). Свой лист в сводке скрыт.
- Тест `snapshot_visibility_matrix_and_replay` (2 клиента): party→оба, gm→только ГМ, hidden→снятие, догон позднего.
  `cargo test party` — **5/5**; `tsc`/`build`/purity/lint — зелёные.

**Как сделано (LP2, часть 2 — полный read-only лист):**
- `CharacterSheet` получил проп `readOnly`. Глушитель мутаций — ОДНА точка: `onUpdate` гейтится в no-op
  (`const onUpdate = readOnly ? NOOP_UPDATE : rawOnUpdate`), и т.к. все правки идут через него, лист неизменяем целиком.
- Контролы НЕ блокируются целиком: иначе нельзя раскрыть способности/чувства/скорости (всё это по клику). Безопасность —
  единственно через глушитель `onUpdate` (правки не сохраняются и до владельца не доходят; снимки односторонние). Скрыты
  лишь явные точки редактирования: кнопка левел-апа, FullEditPanel, гейт `onPortraitClick`. Сверху — баннер «Только чтение».
  (Пробовали `<fieldset disabled>` — откатили: глушил и просмотр описаний/раскрытие разделов. Визуальное гашение/скрытие
  именно мутационных контролов — кандидат в дизайн-проход LPD.)
- Открытие: карточка в сводке ГМа кликабельна → полноэкранный оверлей `ReadOnlySheetOverlay` с `CharacterSheet readOnly`
  (данные = `Character` из снимка), закрытие по X / Back / Esc (`useBackDismiss`). Превью своего листа в ShareControls —
  не кликабельно. `PartyPanel` статически импортит `CharacterSheet` (шарится с основным бандлом; в веб не тянется — online-чанк там не грузится).
- Проверки: `tsc`/`build`/purity зелёные; lint новых правок чист. Live-проверку оверлея делает пользователь.
- **Пост-фиксы read-only по фидбеку (2026-06-16):** **живое обновление** открытого листа — оверлей хранит ID участника и
  берёт снимок из `snapshots` на каждом рендере (правка владельца → лист обновляется без переоткрытия, вкладка просмотра
  сохраняется); авто-закрытие, если владелец перестал делиться. Подход к блокировке переосмыслен: `fieldset disabled`
  (+pre-expand способностей +курсор «нельзя») откатили, т.к. он не давал смотреть способности/чувства/скорости —
  теперь read-only = просмотр работает везде, мутации глушит только `onUpdate`.

### LP3 — игровой лог 🟡 ядро (транспорт + броски + лента + приватный тумблер) сделано; осталось события листа, фильтры, экспорт
1. `src/online/gameLog.ts`: `logEvent(type, payload)` — шлёт `event` в партию; вне партии/сети — no-op.
2. Эмит из существующих точек: броски (`DiceRollProvider`/`DiceTab`), события листа (урон/лечение, отдыхи, левел-ап,
   ресурсы) — из явных обработчиков, НЕ диффом состояния. Тумблер «приватный бросок».
3. Лента на экране партии (живая, с пагинацией назад, фильтры по типу/персонажу, локализованный рендер).
4. Экспорт ленты в `.txt`/`.json`.
5. Критерий: бросок игрока появляется в ленте у всех мгновенно; приватный — не появляется; экспорт скачивает файл.

**Как сделано (LP3, ядро):**
- **Транспорт:** сообщение `event` (любой→хост) → хост штампует `{id,ts,from,actor,kind,payload}`, кладёт в кольцевой
  буфер (cap 200) и рассылает `party-event` ВСЕМ (включая автора — хост источник порядка) + локально ГМу. Команда
  `party_event`; событие `party://event`; догон лога поздно подключившимся. Тест `event_log_broadcasts_to_all_and_replays`
  (`cargo test party` 7/7).
- **Эмит:** броски — из единой точки `DiceRollProvider.showToast` (и быстрый, и конфиг-бросок). Web-safe шим
  `src/utils/partyLog.ts` (`logPartyEvent`, no-op вне Tauri/партии) лениво грузит `src/online/gameLog.ts`; точки эмита в
  общих компонентах не тянут online статически. **Тумблер «приватный бросок»** — флаг в `partyLog.ts`, проверяется на эмите.
- **Лента:** копится на уровне App (`partyLog`, переживает уход с экрана; дедуп по id), рисуется в `PartyPanel` (GameLogFeed):
  время + актор + `expr = total` (+ пре/пом), авто-скролл, тумблер приватных.
- **Осталось в LP3:** эмит событий листа (HP/отдых/левел-ап/ресурсы) из обработчиков `CharacterSheet`; фильтры
  по типу/персонажу; экспорт ленты в `.txt`/`.json`.

### LPD — дизайн-проход по экранам партии (решено 2026-06-16: ПОСЛЕ функционала)
Отдельный визуальный проход по всем экранам партии (меню/создать/подключиться, активная сессия, состав, шеринг+видимость,
сводка ГМа, read-only лист, лента лога) до уровня остального приложения. Делается **после LP2.2 + LP3**, когда поверхность
партии стабилизируется — чтобы не перекрашивать ещё меняющийся UI. Сейчас экраны — сознательно утилитарный MVP на токенах
приложения (`glass-panel`, золото). Можно использовать skill `frontend-design`.

### LP4 — после v1 (зафиксировано, не сейчас)
- Авто-обнаружение в реальной LAN (UDP broadcast/mDNS) — бонус (через VPN обычно не работает).
- Переподключение при обрыве, миграция хоста, «зрители».
- **WebRTC** для интернета без VPN (нужен маленький signaling-сервер — возврат к лёгкой инфре).
- Чат/свободный текст + жалобы/модерация (только вместе).

## Риски и тонкие места

- **Read-only лист (LP2) — самый дорогой UI-кусок**: мутации размазаны (HP, инвентарь, слоты, ресурсы, отдыхи,
  трансформации, заметки). Заглушка на `updateCharacter` страхует от пропущенных контролов.
- **Mixed-content**: именно поэтому сеть на Rust, не в JS. Если соблазнимся `new WebSocket` во фронте — упрётся в блок.
- **Файрвол Windows** спросит разрешение на прослушку порта у хоста (один клик «Разрешить»).
- **Надёжность по VPN**: нужен heartbeat/timeout, аккуратная обработка обрывов; хост-дисконнект в v1 = партия завершается.
- **Безопасность**: код партии — деттерент (на общей VPN кто угодно может попытаться подключиться); снимки листов —
  read-only данные, без чувствительного текста (чата нет).
- **Версии протокола**: рукопожатие должно явно отбивать несовместимые версии приложения.
- **Async-сложность в Rust**: держим сообщения простым JSON, состояние партии — в одном акторе/мьютексе.

## Не-цели (v1)

- Центральный сервер, аккаунты, облачная персистентность (это отменённый VPS-план).
- Интернет без VPN (WebRTC/signaling) — post-v1.
- Чат, шёпоты, тайные броски «только ГМ видит», совместный боевой трекер.
- Подмена локального хранилища: персонажи остаются у каждого локально, по сети идут только снимки.

## Параметры (решено 2026-06-16)

- **Без лимита участников.**
- **Порт: фиксированный редкий по умолчанию (`47331`), не случайный.** Если при старте хоста занят — авто-фоллбэк
  на ближайший свободный; фактический `IP:порт` всё равно показывается в UI хоста (его нужно сообщить игрокам).
- Клиент и сервер — на **Rust** (mixed-content, зафиксировано в «Ключевых решениях»).

## Код партии (решено 2026-06-16)

**Одна строка, кодирующая `IP:порт` + секрет.** Игрок вставляет её одну — адрес и пароль внутри, отдельных полей
нет («вставил — подключился»). Хост генерирует код при создании партии (реализация: компактная кодировка
`ip:port|secret`, напр. base64url, с валидацией формата при вставке).

Открытых вопросов под старт LP0 не осталось.
